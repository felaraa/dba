# Architecture Map

Ultima revisao: 2026-06-25.

Este documento descreve a arquitetura do `oracle-query-otim` para agentes e
desenvolvedores. Ele deve ser usado quando a tarefa envolve fluxo entre modulos,
responsabilidades, contratos ou decisao de onde implementar uma mudanca.

## 1. Visao geral

O projeto e um advisor CLI/biblioteca para tuning de queries Oracle 19c RAC. Ele
recebe SQL, plano de execucao, metadados e perfil do ambiente, monta um
`RuleContext` imutavel, executa regras deterministicas e produz recomendacoes
ranqueadas em texto/Markdown. Opcionalmente, em ambiente com banco real, coleta
SQL/plano/metadados por SQL_ID, analisa top SQLs em lote e valida indices com
indice invisivel.

Arquitetura em alto nivel:

```text
SQL text ---------------------> sql_parser -------> ParsedQuery
Plan XML/text ----------------> plan_parser ------> ParsedPlan
Oracle DB or fixture ---------> metadata_collector -> SchemaMetadata
env_profile YAML -------------> env_profile ------> EnvProfile
SQL_ID history, DB mode only -> plan_history -----> PlanHistory

ParsedQuery + ParsedPlan + SchemaMetadata + EnvProfile + PlanHistory
  -> RuleContext
  -> RuleEngine
  -> Recommendation[]
  -> reporter.consolidate_indexes
  -> reporter.merge_mitigation_warnings
  -> reporter.to_text / reporter.to_markdown

Optional:
  -> validator.InvisibleIndexValidator for --validate
```

## 4. Core contracts

Main file: `src/advisor/models.py`.

Important contracts:
- `ParsedQuery`: raw SQL, table refs, joins, filters, projected columns and
  group-by columns.
- `ParsedPlan`: SQL_ID, plan hash, operations, total metrics, plan notes and
  SQL Profile.
- `PlanOperation`: operation id/name/object, E-Rows/A-Rows, executions,
  predicates, parent id, owner and workarea metrics.
- `SchemaMetadata`: tables, columns and indexes collected from Oracle or fixture.
- `PlanHistory`: plan stats by plan hash for one SQL_ID.
- `Recommendation`: rule output with severity, rationale, optional DDL,
  target table, scores, tags and warnings.

Boundary rule:
- If a change adds data needed by several rules, add it to these contracts and
  update parsers/collectors/tests together.
- If a change is only presentation, keep it in `reporter.py`.
- If a change is only one diagnosis, keep it in a rule plugin.

## 5. Rule architecture

Files:
- `src/advisor/rule_base.py`
- `src/advisor/engine.py`
- `src/advisor/rules/`

Design:
- Rules subclass `Rule`.
- Each rule implements `evaluate(ctx: RuleContext) -> list[Recommendation]`.
- `RuleContext` is immutable and contains query, plan, metadata, env profile and
  optional plan history.
- `RuleEngine` imports all modules in `advisor.rules`, instantiates `Rule`
  subclasses, filters by allow/deny list and sorts by numeric `priority`.
- Rules do not communicate with each other.
- Rule failures are logged and isolated.

Priority convention:
- `1-9`: context/diagnostic rules that should precede index recommendations.
- `10-40`: rules that generate index recommendations or access-path findings.
- `900+`: mitigation rules such as RAC hot block warnings.

Current rule order:

| Priority | Rule | Role |
|---:|---|---|
| 1 | R005 existing intervention | SQL Profile/Baseline/Outline context |
| 2 | R009 plan instability | multiple plan hashes |
| 5 | R004 cartesian/bad estimates | cardinality/statistics diagnosis |
| 6 | R010 workarea spill | TEMP/workarea diagnosis |
| 7 | R011 massive rowid access | index used for too many rows |
| 8 | R007 unused existing index | index exists but optimizer ignores it |
| 10 | R001 filter should be access | missing probe index |
| 20 | R002 avoidable full scan | selective join full scan |
| 25 | R006 buffer sort materialization | materialization for join |
| 30 | R003 covering aggregation | covering index |
| 40 | R008 global index on partitioned | maintenance debt alert |
| 900 | R900 RAC hotblock mitigation | RAC mitigation warnings |

## 6. Index and DDL architecture

Shared helpers live in `src/advisor/rules/__init__.py`.

Mandatory path for index-generating rules:
1. Resolve owner with `ctx.resolve_owner(table_name, query_owner)`.
2. Decide locality with `ctx.is_partitioned(owner, table_name)`.
3. Check existing indexes with `existing_index_covering` or
   `existing_index_exact_or_superset`.
4. Order columns with `order_columns(equality, range_cols, covering)`.
5. Name index with `build_index_name(table, columns, owner=owner)`.
6. Build DDL with `build_index_ddl(owner, table, index_name, columns, local,
   parallel=ctx.env.index_parallel, tablespace=ctx.env.index_tablespace)`.

DDL properties:
- Owner-qualified index and table names.
- `LOCAL` when table is partitioned.
- Optional `TABLESPACE`.
- Optional `PARALLEL n` for creation plus `ALTER INDEX ... NOPARALLEL`.
- `DBMS_STATS.GATHER_INDEX_STATS` after create.

Do not:
- Generate `None.TABLE` or unqualified owner DDL.
- Recommend an index already covered by existing metadata.
- Deduplicate inside rules; reporter owns consolidation.

## 7. Parser architecture

### SQL parser

File: `src/advisor/sql_parser.py`.

Responsibilities:
- Normalize Oracle numeric binds for `sqlglot`.
- Extract table refs with owner/name/alias.
- Extract equality joins between aliases.
- Extract equality/range/IN filters.
- Extract projected and group-by columns from all SELECT nodes.

Limit:
- Ambiguous unqualified columns are ignored.

### Plan parser

File: `src/advisor/plan_parser.py`.

Responsibilities:
- Auto-detect SQL Monitor XML vs DBMS_XPLAN text.
- SQL Monitor XML:
  - parse static plan operation names, objects, predicates and E-Rows;
  - parse runtime parent ids, starts, A-Rows, read bytes, owner and workarea
    metrics;
  - detect SQL Profile/Baseline/Outline notes.
- DBMS_XPLAN text:
  - reconstruct hierarchy from indentation;
  - parse E-Rows, A-Rows, Starts, Buffers;
  - attach Predicate Information to operations.

Important:
- XML is preferred because it provides richer runtime metrics.
- Text remains fallback for environments without SQL Monitor.

## 8. Metadata architecture

File: `src/advisor/metadata_collector.py`.

Implementations:
- `OracleMetadataCollector`: reads live metadata from Oracle DBA views.
- `FixtureMetadataCollector`: returns prebuilt `SchemaMetadata` for tests/offline.

Oracle collection:
- Tables/views and partition flags.
- Partition key columns.
- Column stats.
- Indexes, index columns, partition/locality.
- Index usage.
- Stats health (`last_analyzed`, stale stats, stale partitions).

Resilience:
- One table/view failure should not abort the whole pipeline.
- Missing tables are reported through `collector.missing`.
- Rules must handle incomplete metadata defensively.

Important bug-avoidance:
- `_indexes` fetches all index rows before nested cursor queries. Do not replace
  this with lazy cursor iteration.

## 9. Environment and AWR architecture

Files:
- `src/advisor/env_profile.py`
- `src/advisor/awr_parser.py`
- `src/advisor/profile_builder.py`
- `src/advisor/awr_cli.py`
- `config/env_profile_*.yaml`

Principles:
- Environment calibration is data, not code.
- AWR parser extracts facts.
- Profile builder applies policy, defaults and preservation.
- AWR CLI orchestrates create/update.

Create/update flow:

```text
AWR HTML(s)
  -> awr_parser.parse_awr
  -> AwrMetrics
  -> aggregate_metrics if multiple AWRs
  -> profile_builder.build_profile(existing optional)
  -> profile_builder.emit_yaml
  -> config/env_profile_<name>.yaml
```

Human fields preserved on update:
- `scoring.*`
- `index_ddl.*`
- `identity.exadata`
- `io.full_scan_block_discount`
- `workload.benefit_metric`

AWR-derived fields refreshed:
- identity except Exadata where applicable;
- workload facts;
- IO facts;
- RAC contention and hot segments when present;
- optimizer facts.

## 10. CLI architecture

### Single-query CLI

File: `src/advisor/cli.py`.

Modes:
- File mode: `--sql <file> --plan <file>`.
- SQL_ID mode: `--sql-id <id> --source <db>`.
- Fixture mode: `--source fixture --fixture tests.fixtures_*`.
- DB mode: `--source rawdb|datadb|db|...`.

Flow:
1. Load env profile.
2. Read SQL/plan from files or fetch by SQL_ID.
3. Parse SQL and plan.
4. Load fixture metadata or connect and collect Oracle metadata.
5. Collect plan history in DB mode.
6. Build `RuleContext`.
7. Run `RuleEngine`.
8. Consolidate indexes and merge mitigations.
9. Render text/Markdown.
10. Optionally validate with invisible indexes.

### Batch CLI

Files:
- `src/advisor/batch.py`
- `src/advisor/batch_cli.py`

Flow:
1. Query top SQL_IDs from `GV$SQL`.
2. Fetch SQL text from `GV$SQLTEXT`.
3. Fetch plan by SQL Monitor XML, fallback DBMS_XPLAN.
4. Save artifacts under `temp/<sql_id>/`.
5. Run the same parser/metadata/rule/reporter path for each SQL_ID.
6. Emit consolidated batch report.

Constraint:
- Batch requires DB source; fixture is not supported.

### AWR CLI

File: `src/advisor/awr_cli.py`.

Flow:
- Read one or more AWR HTMLs.
- Create or update env profile.
- Print diagnostics about found/missing fields when `--diag`.

## 11. Reporting and validation architecture

### Reporter

File: `src/advisor/reporter.py`.

Responsibilities:
- `consolidate_indexes`: remove redundant index recommendations on same table.
- `merge_mitigation_warnings`: attach warning-only mitigation recs to matching
  index recommendations by target table.
- `to_text` and `to_markdown`: final single-query output.

Batch-specific formatting lives in `src/advisor/batch_cli.py`.

### Validator

File: `src/advisor/validator.py`.

Responsibilities:
- Execute baseline query with `GATHER_PLAN_STATISTICS`.
- Create candidate index as `INVISIBLE`.
- Enable `optimizer_use_invisible_indexes` in the session.
- Re-run query, capture plan and compare buffer gets/elapsed.
- Drop the temporary index in cleanup.

Constraint:
- Validation creates real database objects and must remain opt-in.

## 12. Test architecture

Test root: `tests/`.

Patterns:
- Unit/regression tests run offline.
- Fixtures provide metadata without Oracle.
- Example SQL/plans under `examples/` capture real cases.
- Integration tests that need Oracle must use `@pytest.mark.integration` and
  skip without `ORACLE_*`.

Important tests:
- `tests/test_pipeline.py`: baseline parser/rule pipeline.
- `tests/test_cartesian_case.py`: cartesiano and existing intervention.
- `tests/test_improvements_v3.py`: stale stats, local inference, DDL stats.
- `tests/test_index_collection_fixes.py`: metadata/index collection bugs.
- `tests/test_unused_index_case.py`: existing index ignored.
- `tests/test_plan_instability.py`: plan history and R009.
- `tests/test_workarea_spill.py`: R010.
- `tests/test_massive_rowid_access.py`: R011.
- `tests/test_owner_resolution.py`: owner fallback.
- `tests/test_index_ddl_options.py`: DDL options.
- `tests/test_global_index_rule.py`: R008.
- `tests/test_awr_profile.py`: AWR/profile create/update.

Command:

```bash
python -m pytest -q
```

## 15. Common change paths

Add a new tuning rule:
1. Read `.ai/context-index.md` > New Or Changed Tuning Rule.
2. Read similar rule and test.
3. Add `src/advisor/rules/rule_<name>.py`.
4. Add fixture/examples/test.
5. Update `.ai/implemented-index.md`.
6. Update `.ai/decision-log.md` only if design/priority policy changes.

Add a new field to plan operations:
1. Update `models.py`.
2. Update `plan_parser.py`.
3. Update affected rules.
4. Add parser tests.
5. Run full suite.

Add AWR/env profile support:
1. Parse raw metric in `awr_parser.py`.
2. Apply policy/preservation in `profile_builder.py`.
3. Update `docs/GUIA_ENV_PROFILE.md` if user-visible.
4. Update `tests/test_awr_profile.py`.
5. Record decision if field is human vs AWR-derived.

Change CLI behavior:
1. Update `cli.py` or `batch_cli.py`.
2. Update README/manual if command public surface changes.
3. Add focused tests where feasible.
4. Update implemented index and decision log if contract/policy changed.

## 16. Architectural anti-patterns

Avoid:
- Putting tuning logic in `engine.py`.
- Letting rules depend on other rules' outputs.
- Generating DDL without owner resolution.
- Assuming missing metadata means non-partitioned.
- Hardcoding RAWDB/DATADB exceptions in rules.
- Running DB/validation flows implicitly.
- Adding dependencies for simple parsing/formatting already handled locally.
- Expanding boot files into full project manuals.
- Updating memory for trivial typo-only changes.
